This document covers the three interconnected async systems that power the platform behind every user action: the **Workflow Engine** (business process automation), the **Event Bus** (system-wide event propagation), and the **Job Queue** (background task execution). These three systems are the nervous system of the platform — if the agent layer is the brain, these are the signals moving through the body.

## How The Three Systems Interconnect

```
User/Agent Action
       │
       ▼
┌──────────────┐     emits      ┌──────────────┐
│  Application │ ──────────────►│  Event Bus   │
│  Service     │                │  (Redis      │
│              │                │   Streams)   │
└──────┬───────┘                └──────┬───────┘
       │                               │
       │ may trigger                   │ consumed by
       ▼                               ▼
┌──────────────┐  creates jobs  ┌──────────────┐
│  Workflow    │ ──────────────►│  Job Queue   │
│  Engine      │                │  (BullMQ /   │
│  (state      │◄───────────────│   Redis)     │
│   machine)   │  job complete  │              │
└──────────────┘   triggers     └──────┬───────┘
       │           next step           │
       │                               │ processed by
       │ emits events                  ▼
       │ at each step          ┌──────────────┐
       └──────────────────────►│  Workers     │
                               │  (Node.js +  │
                               │   Python)    │
                               └──────────────┘
```

**The cycle**: A user action creates a record → the application service emits an event → the event bus delivers it to all subscribers → one subscriber is the workflow engine which evaluates if a workflow should start or advance → the workflow engine creates jobs for each step → workers execute jobs → jobs emit completion events → the workflow engine advances to the next step → events emitted at every step feed audit trail, notifications, analytics, and integrations.

## Part 1: Event Bus Architecture

### Technology: Redis Streams

Redis Streams provides ordered, persistent, multi-consumer message delivery. Every event is written once and consumed by multiple independent consumer groups — each consumer group processes the event independently and at its own pace.

### Topology

```
Producer (any service)
    │
    ▼
Redis Stream: "events:{org_id}"
    │
    ├──► Consumer Group: "audit"
    │       └── Worker: writes to audit_events table (PostgreSQL)
    │
    ├──► Consumer Group: "notifications"
    │       └── Worker: evaluates notification rules → sends via channel
    │
    ├──► Consumer Group: "workflows"
    │       └── Worker: evaluates if event should trigger/advance a workflow
    │
    ├──► Consumer Group: "analytics"
    │       └── Worker: writes to ClickHouse for analytical queries
    │
    ├──► Consumer Group: "integrations"
    │       └── Worker: syncs data to CRM/ERP based on event type
    │
    └──► Consumer Group: "search"
            └── Worker: updates Elasticsearch/vector indexes
```

### Event Publishing

Every service publishes events through a shared `EventBus` class. Events are always published AFTER the database transaction commits — never inside the transaction (to avoid phantom events if the transaction rolls back).

```typescript
// lib/event-bus.ts

import Redis from "ioredis";

interface DomainEvent {
  event_id: string;          // UUID, generated by publisher
  event_type: string;        // e.g., "contract.created"
  timestamp: string;         // ISO 8601
  org_id: string;
  actor: {
    type: "user" | "agent" | "system";
    id: string;
    triggered_by_user_id?: string;  // if agent, who triggered it
  };
  resource: {
    type: string;            // e.g., "contract"
    id: string;
  };
  data: Record<string, unknown>;  // event-specific payload
  metadata?: {
    correlation_id?: string;  // traces a chain of related events
    causation_id?: string;    // the event that caused this event
  };
}

class EventBus {
  private redis: Redis;

  async publish(event: DomainEvent): Promise<void> {
    const streamKey = `events:${event.org_id}`;
    await this.redis.xadd(
      streamKey,
      "*",  // auto-generate stream ID
      "payload", JSON.stringify(event)
    );
  }

  // Publish multiple events atomically (for multi-step operations)
  async publishBatch(events: DomainEvent[]): Promise<void> {
    const pipeline = this.redis.pipeline();
    for (const event of events) {
      pipeline.xadd(
        `events:${event.org_id}`,
        "*",
        "payload", JSON.stringify(event)
      );
    }
    await pipeline.exec();
  }
}
```

### Event Consumption

Each consumer group has dedicated workers that read events, process them, and acknowledge completion. Failed events are retried automatically. Events that fail after max retries go to a dead letter stream.

```typescript
// workers/event-consumer.ts

class EventConsumer {
  private group: string;       // e.g., "audit"
  private consumer: string;    // e.g., "audit-worker-1"
  private handler: (event: DomainEvent) => Promise<void>;
  private maxRetries = 5;
  private retryDelayMs = [1000, 5000, 30000, 120000, 600000]; // exponential backoff

  async start(): Promise<void> {
    // Ensure consumer group exists
    try {
      await this.redis.xgroup("CREATE", this.streamKey, this.group, "0", "MKSTREAM");
    } catch (e) { /* group already exists */ }

    // Read loop
    while (true) {
      // Read new events (block for up to 5 seconds)
      const results = await this.redis.xreadgroup(
        "GROUP", this.group, this.consumer,
        "COUNT", 10,
        "BLOCK", 5000,
        "STREAMS", this.streamKey, ">"
      );

      if (!results) continue;

      for (const [streamKey, entries] of results) {
        for (const [id, fields] of entries) {
          const event = JSON.parse(fields[1]) as DomainEvent;
          try {
            await this.handler(event);
            await this.redis.xack(streamKey, this.group, id);
          } catch (error) {
            await this.handleFailure(streamKey, id, event, error);
          }
        }
      }
    }
  }

  private async handleFailure(
    streamKey: string, id: string,
    event: DomainEvent, error: unknown
  ): Promise<void> {
    // Check retry count from pending entries
    const pending = await this.redis.xpending(
      streamKey, this.group, id, id, 1
    );
    const deliveryCount = pending?.[0]?.[3] ?? 0;

    if (deliveryCount >= this.maxRetries) {
      // Move to dead letter stream
      await this.redis.xadd(
        `dead-letter:${this.group}`,
        "*",
        "original_id", id,
        "payload", JSON.stringify(event),
        "error", String(error),
        "failed_at", new Date().toISOString()
      );
      // Acknowledge so it stops retrying
      await this.redis.xack(streamKey, this.group, id);
      // Alert ops
      await this.alertDeadLetter(event, error);
    }
    // Otherwise: leave unacknowledged. Redis will redeliver after claim timeout.
  }
}
```

### Dead Letter Handling

Events that fail all retries land in `dead-letter:{consumer_group}` streams. These are monitored and alerted on.

```
dead-letter:audit          → Alert: audit event failed to persist
dead-letter:notifications  → Alert: notification delivery failed
dead-letter:workflows      → Alert: workflow trigger failed (may need manual intervention)
dead-letter:integrations   → Alert: CRM/ERP sync failed (will retry on next scheduled sync)
dead-letter:search         → Alert: search index update failed (will self-heal on next reindex)
```

Admin dashboard (SCR-041) shows dead letter counts. Ops can replay dead letter events via admin API.

### Event Ordering Guarantees

- **Within a single org stream**: events are strictly ordered (Redis Stream guarantee).
- **Across org streams**: no ordering guarantee (and none needed — orgs are isolated).
- **Within a consumer group**: events are processed in order per consumer. Multiple consumers in a group can process in parallel (events distributed round-robin), but a single entity's events may be processed out of order if they're assigned to different consumers. For critical ordering (e.g., contract version sequence), use the `correlation_id` and handle idempotently.

### Stream Retention

```
events:{org_id}     → MAXLEN ~100000 (trim to last 100K events per org)
dead-letter:*       → MAXLEN ~10000 (keep for investigation, alert on growth)
```

Audit events are persisted to PostgreSQL by the audit consumer — the Redis stream is ephemeral delivery, not the permanent store.

## Part 2: Workflow Engine

### What It Does

The workflow engine manages **business process automation** — approval chains, contract routing, obligation escalation, renewal processes. It is NOT the agent orchestrator (that's LangGraph). The workflow engine handles deterministic, rule-based business processes. The agent orchestrator handles AI-driven, non-deterministic tasks.

| Workflow Engine | Agent Orchestrator |
|----------------|-------------------|
| Deterministic rules | AI reasoning |
| Configured by admin in UI (SCR-035) | Configured by system prompts |
| Approval routing, escalation, notifications | Draft generation, risk analysis, counter-proposals |
| State persisted in PostgreSQL | State persisted in Redis/LangGraph |
| Evaluated on events | Evaluated on user intent |

### Workflow Execution Model

A workflow is a **directed graph** stored in `workflow_definitions` / `workflow_nodes` / `workflow_edges`. When a workflow is triggered, the engine creates an `approval_instance` and walks the graph.

```
                    ┌─────────┐
                    │  START  │
                    └────┬────┘
                         │
                    ┌────▼────┐
                    │CONDITION│──── value > $1M? ────┐
                    └────┬────┘                      │
                    NO   │                      YES  │
                    ┌────▼────┐              ┌───────▼───────┐
                    │APPROVAL │              │   PARALLEL    │
                    │Legal Mgr│              │ ┌───────────┐ │
                    └────┬────┘              │ │VP Legal   │ │
                         │                   │ │           │ │
                         │                   │ │CFO        │ │

                         │                   │   APPROVAL    │

                    │   "Contract approved"            │
                    └────────────┬─────────────────────┘

                            ┌────▼────┐
                            │   END   │
                            └─────────┘
```

### State Machine

Each `approval_instance` has a state machine tracking its progress through the graph:

```typescript
// workflow-engine/state-machine.ts

interface WorkflowState {
  instance_id: string;
  workflow_id: string;
  contract_id: string;
  org_id: string;
  current_nodes: string[];    // Can be multiple for parallel execution
  completed_nodes: string[];
  status: "running" | "paused" | "completed" | "cancelled" | "failed";
  context: Record<string, unknown>;  // Data accumulated during execution
  started_at: string;
  updated_at: string;
}

class WorkflowStateMachine {

  // Evaluate which node(s) to execute next
  async advance(state: WorkflowState): Promise<void> {
    for (const nodeId of state.current_nodes) {
      const node = await this.getNode(nodeId);

      switch (node.node_type) {
        case "start":
          await this.moveToNextNodes(state, nodeId);
          break;

        case "condition":
          const result = await this.evaluateCondition(node, state.context);
          // Follow the edge whose condition matches
          const matchingEdge = await this.findMatchingEdge(nodeId, result);
          await this.moveToNode(state, matchingEdge.target_node_id);
          break;

        case "approval":
          // Create approval step, wait for human decision
          await this.createApprovalStep(state, node);
          // State pauses here — resumes when approval decision arrives via event
          break;

        case "parallel":
          // Fork: activate all child paths simultaneously
          const childEdges = await this.getOutgoingEdges(nodeId);
          const childNodeIds = childEdges.map(e => e.target_node_id);
          state.current_nodes = childNodeIds;
          await this.advance(state);  // recurse into each child
          break;

        case "notification":
          await this.sendNotification(node, state.context);
          await this.moveToNextNodes(state, nodeId);
          break;

        case "timer":
          // Schedule a delayed job
          await this.scheduleTimer(state, node);
          // State pauses — resumes when timer fires
          break;

        case "escalation":
          await this.escalate(state, node);
          await this.moveToNextNodes(state, nodeId);
          break;

        case "end":
          state.status = "completed";
          await this.onComplete(state);
          break;
      }
    }

    await this.persistState(state);
  }

  // Called when an approval decision event arrives
  async onApprovalDecision(
    instanceId: string,
    stepId: string,
    decision: "approved" | "rejected"
  ): Promise<void> {
    const state = await this.loadState(instanceId);

    if (decision === "rejected") {
      state.status = "failed";
      await this.onRejected(state, stepId);
      return;
    }

    // Check if this was a parallel step — need all parallel siblings to complete
    const node = await this.getNodeForStep(stepId);
    if (await this.isParallelNode(node)) {
      const allSiblings = await this.getParallelSiblings(state, node);
      const allApproved = allSiblings.every(s => s.status === "approved");
      if (!allApproved) return;  // Wait for remaining siblings
    }

    // Move to next node(s)
    await this.moveToNextNodes(state, node.id);
    await this.advance(state);
  }

  // Condition evaluation
  private async evaluateCondition(
    node: WorkflowNode,
    context: Record<string, unknown>
  ): Promise<boolean> {
    const config = node.config as ConditionConfig;
    // config.expression = { field: "contract.value", operator: ">", value: 1000000 }
    const fieldValue = this.resolveField(config.expression.field, context);

    switch (config.expression.operator) {
      case ">": return fieldValue > config.expression.value;
      case "<": return fieldValue < config.expression.value;
      case "==": return fieldValue === config.expression.value;
      case "in": return config.expression.value.includes(fieldValue);
      case "contains": return String(fieldValue).includes(String(config.expression.value));
      default: return false;
    }
  }
}
```

### Workflow Triggers

Workflows are triggered by events. The workflow consumer listens for events and checks if any active workflow definition matches:

```typescript
// workers/workflow-trigger.ts

class WorkflowTriggerConsumer {
  async handleEvent(event: DomainEvent): Promise<void> {
    // Find workflow definitions that match this event
    const workflows = await this.findMatchingWorkflows(event);

    for (const workflow of workflows) {
      // Check: does this contract match the workflow's "applies_to" criteria?
      const contract = await this.contractService.get(event.resource.id);
      if (!this.matchesAppliesTo(workflow, contract)) continue;

      // Check: is there already a running instance for this contract + workflow?
      const existing = await this.findExistingInstance(contract.id, workflow.id);
      if (existing && existing.status === "running") continue;  // Don't double-trigger

      // Create new workflow instance and start execution
      const instance = await this.createInstance(workflow, contract, event);
      const engine = new WorkflowStateMachine();
      await engine.advance(instance);
    }
  }
}

// Trigger mapping: which events can start which workflow types
```

### Timer and Escalation Mechanics

Timers are implemented as delayed BullMQ jobs. When a timer node is reached, the engine creates a job with the specified delay. When the job fires, it emits an event that the workflow engine picks up to advance.

```typescript
// Escalation example: if approver doesn't respond in 48 hours
async scheduleTimer(state: WorkflowState, node: WorkflowNode): Promise<void> {
  const config = node.config as TimerConfig;
  const delayMs = config.duration_hours * 60 * 60 * 1000;

  await this.jobQueue.add(
    "workflow.timer_fired",
    {
      instance_id: state.instance_id,
      node_id: node.id,
      action: config.on_timeout,  // "escalate" | "auto_approve" | "cancel"
    },
    {
      delay: delayMs,
      jobId: `timer:${state.instance_id}:${node.id}`,  // Idempotent: same key = no duplicate
      removeOnComplete: true,
    }
  );
}

// When the timer fires (in worker):
async handleTimerFired(job: Job): Promise<void> {
  const { instance_id, node_id, action } = job.data;
  const state = await this.loadState(instance_id);

  // Check if the step was already completed (approved before timer fired)
  if (state.completed_nodes.includes(node_id)) return;  // No-op: already resolved

  switch (action) {
    case "escalate":
      // Find the pending approval step, reassign to escalation target
      await this.reassignToEscalation(state, node_id);
      break;
    case "auto_approve":
      // Auto-approve and advance
      await this.autoApprove(state, node_id);
      break;
    case "cancel":
      state.status = "cancelled";
      await this.persistState(state);
      break;
  }
}
```

## Part 3: Job Queue (Background Tasks)

### Technology: BullMQ (Redis-backed)

BullMQ provides reliable job queues with priorities, retries, delayed jobs, rate limiting, and progress tracking. Every long-running or non-blocking operation runs as a background job.

### Job Failure Handling

```typescript
// Global failure handler for all queues
worker.on("failed", async (job, error) => {
  if (job.attemptsMade >= job.opts.attempts) {
    // All retries exhausted — move to dead letter
    await deadLetterQueue.add(job.name, {
      original_job: job.toJSON(),
      error: error.message,
      stack: error.stack,
      failed_at: new Date().toISOString(),
    });

    // Alert based on job criticality
    if (job.queue.name === "agent-tasks") {
      await alerting.critical(`Agent task failed: ${job.name}`, { jobId: job.id, error });
    } else if (job.queue.name === "workflow-tasks") {
      await alerting.high(`Workflow task failed: ${job.name}`, { jobId: job.id, error });
    } else {
      await alerting.medium(`Background task failed: ${job.name}`, { jobId: job.id, error });
    }
  }
});
```

### Rate Limiting for LLM Calls

LLM API calls are expensive and rate-limited by providers. All agent tasks go through a shared rate limiter:

```typescript
// lib/llm-rate-limiter.ts

class LLMRateLimiter {
  // Per-org limits (configurable per subscription tier)
  private limits = {
    free:       { rpm: 10,  tpm: 100_000 },
    pro:        { rpm: 60,  tpm: 1_000_000 },
    enterprise: { rpm: 300, tpm: 10_000_000 },
  };

  async acquireToken(orgId: string, estimatedTokens: number): Promise<boolean> {
    const tier = await this.getOrgTier(orgId);
    const limit = this.limits[tier];

    // Check requests-per-minute
    const rpmKey = `llm:rpm:${orgId}`;
    const currentRPM = await this.redis.incr(rpmKey);
    if (currentRPM === 1) await this.redis.expire(rpmKey, 60);
    if (currentRPM > limit.rpm) return false;

    // Check tokens-per-minute
    const tpmKey = `llm:tpm:${orgId}`;
    const currentTPM = await this.redis.incrby(tpmKey, estimatedTokens);
    if (currentTPM === estimatedTokens) await this.redis.expire(tpmKey, 60);
    if (currentTPM > limit.tpm) return false;

    return true;
  }
}
```

## Part 4: Scheduled Tasks Detail

### Cron Schedule

| Task | Schedule | What It Does | Failure Impact |
|------|----------|-------------|----------------|
| `obligation_check` | Every hour | Scan all obligations → update status (pending→approaching→overdue) → send alerts for approaching/overdue | Missed obligation alerts (up to 1 hour delay) |
| `renewal_scan` | Daily 6am UTC | Scan contracts with expiry in next 90/60/30 days → create renewal alerts → assign to owners | Missed renewal notification (up to 24h delay) |
| `analytics_refresh` | Every hour | Refresh ClickHouse materialized views for dashboard metrics | Stale dashboard data (up to 1h) |
| `stale_signature_check` | Every 6 hours | Find signature requests with no activity in 48h+ → send reminders → escalate | Stalled signatures not followed up |
| `integration_health` | Every 15 min | Ping all active integrations → update health status → alert on failures | Undetected integration outage |
| `dead_letter_alert` | Every hour | Count dead letter entries across all queues → alert if any new | Unprocessed failures not investigated |
| `stream_trimming` | Daily 3am UTC | Trim Redis streams to MAXLEN → clean up old events | Redis memory growth |
| `report_delivery` | Per schedule | Generate and deliver scheduled reports | Reports not delivered on time |
| `backup_verification` | Daily 2am UTC | Verify latest backup integrity → alert if invalid | Undetected backup failure |

### Idempotency

Every scheduled task must be idempotent — running it twice produces the same result as running it once. This is critical because:
- Cron may fire the same job twice (clock skew, restart)
- A failed and retried job runs the same logic again

```typescript
// Example: obligation_check is idempotent
async function checkObligations(orgId: string): Promise<void> {
  const now = new Date();

  // Update statuses based on current date (idempotent: same input = same output)
  await db.obligations.updateMany({
    where: { org_id: orgId, status: "pending", due_date: { lte: addDays(now, 30) } },
    data: { status: "approaching" },
  });

  await db.obligations.updateMany({
    where: { org_id: orgId, status: { in: ["pending", "approaching"] }, due_date: { lt: now } },
    data: { status: "overdue" },
  });

  // Send alerts (use deduplication key to avoid duplicate notifications)
  const approaching = await db.obligations.findMany({
    where: { org_id: orgId, status: "approaching" },
  });

  for (const obligation of approaching) {
    await notificationService.sendIfNotAlreadySent({
      dedup_key: `obligation:approaching:${obligation.id}:${formatDate(now)}`,
      // ↑ Same obligation + same day = no duplicate notification
      recipient: obligation.owner_id,
      type: "obligation_approaching",
      data: { obligation },
    });
  }
}
```

### Overlap Prevention

BullMQ's `jobId` parameter prevents duplicate scheduled jobs:

```typescript
// Cron scheduler (runs in a single process)
import { Queue } from "bullmq";

const scheduledQueue = new Queue("scheduled-tasks", { connection: redis });

// Add repeatable jobs (BullMQ handles scheduling)
await scheduledQueue.add("obligation_check", {}, {
  repeat: { every: 3_600_000 },  // Every hour
  jobId: "obligation_check",      // Prevents duplicates
});

await scheduledQueue.add("renewal_scan", {}, {
  repeat: { pattern: "0 6 * * *" },  // Daily at 6am UTC
  jobId: "renewal_scan",
});
```

## Part 5: Putting It All Together — Example End-to-End Flow

### Scenario: Contract approved → sent for signature → signer doesn't respond → escalation

```
1. Approver clicks "Approve" on SCR-013
   │
   ▼
2. API: POST /approvals/instances/:id/steps/:sid/decide { decision: "approved" }
   │  → approval_steps row updated
   │  → Event emitted: "approval.approved"
   │
   ▼
3. Event Bus delivers "approval.approved" to all consumer groups:
   │
   ├──► Audit consumer: writes to audit_events table
   ├──► Notification consumer: sends "Your contract was approved" to requestor
   ├──► Analytics consumer: writes to ClickHouse
   │
   └──► Workflow consumer: checks if this advances a workflow
        │
        ▼
4. Workflow engine: loads approval_instance state → all steps approved →
   advance to next node → next node is "notification" → sends notification →
   advance to next node → next node is "end" → status = "completed" →
   Event emitted: "approval.workflow_completed"
   │
   ▼
5. Application service: contract status → "approved" →
   Event emitted: "contract.status_changed" { new_status: "approved" }
   │
   ▼
6. Workflow consumer: "contract.status_changed" with status="approved" →
   checks if auto-signature workflow exists → YES →
   creates new workflow instance for signature flow
   │
   ▼
7. Signature workflow starts → first node: "prepare_signature" →
   creates job: agent-tasks / "agent.prepare_signature"
   │
   ▼
8. Agent worker: Signature Agent prepares packet → calls DocuSign API →
   signature sent → Event: "signature.sent"
   │
   ▼
9. Workflow advances to "wait_for_signature" node with TIMER (48 hours)
   │  → Delayed job created: workflow-tasks / "workflow.timer_fired" (delay: 48h)
   │
   │  ... 48 hours pass, signer hasn't responded ...
   │
   ▼
10. Timer fires → Worker handles "workflow.timer_fired" →
    action = "escalate" →
    Creates job: workflow-tasks / "workflow.send_reminder" to signer →
    Creates job: workflow-tasks / "workflow.send_notification" to requestor
    ("Your contract hasn't been signed yet — we've sent a reminder") →
    Schedules new timer for 24 more hours →
    Event: "signature.reminder_sent"
    │
    ▼
11. All events flow through audit, notifications, analytics consumers
    throughout this entire chain. Every step is logged.
```

This is the full cycle: user action → API → event → workflow engine → job queue → worker → event → next workflow step → repeat until done.
