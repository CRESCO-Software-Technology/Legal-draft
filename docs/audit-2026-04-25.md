### F-01 — HeroAgent hidden behind feature flag, off by default
- **Severity:** **P0** — the user's #1 pain point: "I was expecting a Genspark-like primary agent and that agent screens lets you do everything via chat"
- **Screen / Route:** `/dashboard` · `apps/web/src/components/agent/HeroAgent.tsx` line 129
- **Persona affected:** ALL — this is the entire agent identity
- **What I expected:** Login → see prominent chat composer at the top of dashboard, with starter chips that pull from my real queue (per docs/30 §3.1, docs/29 §3 Pattern B+E).
- **What happened:** Logged in fresh as admin, dashboard renders 4 KPI cards + "Recent Activity". **No agent visible.** The HeroAgent component returns `null` because `useFeatureFlag('AGENT_SIDE_PANEL_V2')` resolves to `false` (no localStorage key set).
- **Evidence:**
  - With flag OFF (default): `scripts/audit-screenshots/r3-dashboard.png` — green "you're all caught up" banner only
  - With flag ON: `scripts/audit-screenshots/r3-dashboard-with-hero.png` — the composer + side rail render exactly as intended
- **Suggested fix:** Flip default to enabled OR add an onboarding step that turns the flag on for new orgs. Plan-level: promote the agent surfaces from "experimental" to "default" status. After 6 months of D.x work, the agent isn't even visible — that's a release-blocker.
- **Plan slot:** P7.1 (highest)

## Q.4 — Contract Detail (Zynga MSA — the headline screen)

### F-30 — **CRITICAL** Multi-tenant login routing — same email across orgs lands in wrong org
- **Severity:** **P0** — multi-tenancy security/isolation bug
- **Screen:** `/login` → JWT issuance · `apps/api/src/routes/auth.ts` line 99
- **Persona:** ALL — affects every login
- **What I expected:** Login routes the user to THEIR org; if email is unique by `(orgId, email)`, login should require disambiguation OR use email-globally-unique semantics enforced by DB.
- **What happened:** `prisma.user.findFirst({ where: { email, deletedAt: null } })` — no orgId filter. First match wins (oldest created_at typically). When two demo orgs both have `admin@demo.com`, the older one's user is chosen. JWT issued for the wrong org. Caller sees the wrong org's data.
- **Evidence:** First admin login resolved to `cmmxnzdsf...` org instead of my seeded `cmmx5mzce...`. Maya's contracts list showed "Acme Innovations", "Wayne Enterprises", "Pied Piper" — fictional contracts from a different org I never seeded.
- **Suggested fix (non-trivial):** Either (a) add a global unique constraint `@unique` on `User.email` + migrate to enforce one-user-per-email, or (b) build an org picker into login when email matches multiple users, or (c) require a tenant subdomain (`acme.clm.app/login`) so org is determined by URL. (a) is fastest, (c) is most enterprise-mature.
- **Plan slot:** **P7.0 (must-fix-before-anything)**
